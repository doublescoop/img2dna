img2dna

Encode a black-and-white image as DNA, decode DNA back to an image. Entirely client-side.

W=image width · GR_K=5 · MARKER=TCCG · FSM map · per-strand header: guard(1b) + salt(16b)

FSM map how bits become DNA bases (and back)

Bits are grouped into base-3 digits (trits), and each trit picks the next DNA base according to the previous one. The rule guarantees no base is ever repeated (no AA/CC/GG/TT) and keeps GC% balanced. This exact logic should be used to encode/decode in this project.

previous base trit = 0 trit = 1 trit = 2
A C G T
C G T A
G T A C
T A C G

First base (no previous): trit 0 → A, trit 1 → C, trit 2 → G. To decode, look up the row for the previous base and read the column whose cell equals the current base — that column index is the trit.

decoding steps under the hood

=== Input cleaning ===

  1. Drop FASTA/comment header lines (lines starting with > or #).
  2. Keep only A/C/G/T, uppercase. Everything else (whitespace, punctuation, numbers) is ignored.

=== Split into chunks ===

  1. Split on TCCG. Each piece between markers is one chunk (see chunk boundary).

=== Per-chunk decode ===

  1. DNA → trits using the FSM in reverse.
  2. Trits → bits (base-3 number → binary).
  3. Skip leading zeros; the first 1 is the guard.
  4. Next 16 bits = salt.
  5. XOR the remaining bits with the salt (repeating every 16 bits) to unscramble the body.

=== Rendering ===

  1. Loop until body bits run out: read 1 bit = row's start color; GR-decode runs until they sum to W; paint that row with alternating colors.
  2. Paint rows top-to-bottom in paint order.

Undecodable chunks are skipped. Unpainted pixels stay white.

Decoded image

Rows appear in paint order — the sequence they show up in your pasted text. Partial decoding including at least one chunk starting with the Marker is fine: drop chunks, rearrange, or paste a fragment — only the rows you have will render.

How encoding works

=== Preprocessing ===

  1. Grayscale the image and boost contrast around mid-gray.
  2. Binarize: pixels below threshold → black, above → white.
  3. Downscale to the target width. Every row now has exactly W pixels.

=== Compression ===

  1. RLE: describe each row as "K black, M white, …" — a list of run lengths instead of pixel-by-pixel.
  2. Golomb-Rice: pack each run length into a short prefix code (small runs → few bits).

=== Packaging into chunks ===

  1. Group rows into chunks of roughly target body bits each.
  2. For each chunk, prepend a guard bit + a salt, then XOR the body with the salt (reversible scramble).
  3. Turn those bits into trits (base-3), then into DNA bases through the FSM.

=== Bio-safety check ===

  1. Salt-search: repeat the XOR step with salts 0 … 65,535 until the resulting DNA is bio-safe — GC% within 40–60% and no restriction sites.

=== Output ===

  1. Glue all chunks with the TCCG marker into one DNA string. Also save a FASTA version with per-chunk headers.

Output DNA

The DNA string looks like TCCG…TCCG…TCCG…. Paste it into the Decode tab to verify the round-trip. The W value printed in the status is the width you should use when decoding.

Glossary

Fundamentals (both tabs)

DNA
A molecule that stores information as a sequence of four letters: A, C, G, T. In this app, DNA is just text — you can paste it, copy it, and it'll print the same on any screen.
Base
One letter of DNA: A, C, G, or T. Everything we do here is ultimately a string of bases.
Pixel
One dot in the image. Here each pixel is either black or white (no grays).
Row
One horizontal line of pixels in the image. Every row is exactly W pixels wide.
Bit
A single 0 or 1. Computers store everything as bits.
Trit
Like a bit, but with three possible values instead of two: 0, 1, or 2. We use trits as a stepping stone between bits and DNA bases (since we skip one of the four bases at each position, three choices remain).
FSM map
The lookup table that turns a trit into the next DNA base, given the previous base. Shown as a table at the top of the page. Its job: guarantee no two adjacent bases are equal, which helps keep the DNA balanced and stable.
Marker (TCCG)
A four-letter DNA pattern we insert between chunks so the decoder can find chunk boundaries. TCCG contains CC, which the FSM never produces, so the marker can't accidentally appear inside a chunk. (@Sebastian- let me know if other marking is preferred!)
Chunk
One piece of the image encoded as a short DNA strand. Each chunk carries a group of consecutive rows plus a small header. Chunks are independent — missing or scrambled ones don't corrupt the others.
Strand
A single DNA molecule. In this app, one strand = one chunk. If you ask a lab to synthesize DNA, they'll return one molecule per chunk.
Header
Tiny metadata at the start of each chunk: 1 guard bit + 16 salt bits = 17 bits total. The decoder reads the header before the body to know how to unscramble what follows.
Guard bit
A leading 1 bit placed at the very front of each chunk. Without it, any zero bits at the start would get silently lost when converting between bits and base-3 numbers.
Salt
A 16-bit number stored in the chunk header. The chunk's body is XOR-scrambled with this salt. By trying different salts, we can produce different-looking DNA from the same image data until we find one that is bio-safe.
XOR
A bit-flip operation: 0 XOR 0 = 0, 1 XOR 1 = 0, 0 XOR 1 = 1. Doing it twice returns you to the original — so XOR is a reversible scramble.
DNA string
All chunks' DNA concatenated into one long string, separated by the marker TCCG. Looks like TCCG…TCCG…TCCG…. Easiest format for pasting.
FASTA
The most common text format for DNA in biology. Each record is a header line starting with > (the sequence name) followed by one or more lines of bases. Example: 
>chunk_0 salt=47 rows=9
TCCG…

Decode tab

Image width (W)
How many pixels wide each row is. You must tell the decoder this — otherwise it doesn't know where one row ends and the next begins. The encoder prints the right value to use when it saves the DNA.
Paint order
The decoder paints rows top-to-bottom in the order they appear in your pasted text. If you paste chunks out of order, the image comes out shuffled by chunks (but each chunk's rows stay correct). If you paste them in order, you get the original image.
Partial decoding
You don't need the full the DNA for the whole image. Paste one chunk and you'll see corresponding rows — the rest of the canvas stays white. This is the whole point of using Golomb Rice compression over more effective(e.g., arithmetic encoding) compression methods: you can decode a fragment of the picture. Minimum one chunk including the Marker is required.
Chunk boundary
Where one chunk ends and the next begins. Always marked by TCCG. The decoder splits on this pattern.

Encode tab

Source image
The image you upload — can be any format (PNG, JPG, etc.). It'll be processed into black-and-white before encoding.
Contrast
A multiplier that stretches the image's gray values around the middle. Higher contrast makes darks darker and lights lighter. Useful before thresholding if the source is muddy.
Threshold
The gray value (0–255) below which a pixel becomes black, above which it becomes white. Lower threshold = less black. Tune until the preview looks right.
Binarization
Converting a grayscale image into pure black and white (only two pixel values). Our encoder expects a binary image because runs of all-black or all-white pixels compress very efficiently.
Target width
How wide to resize the image to before encoding. Smaller = less DNA needed (fewer pixels). This becomes the W you'll need on the decode side.
Target body bits
How many bits of compressed image data to put into each chunk's body. Roughly controls DNA strand length — more bits per chunk means fewer, longer strands. 3000 is the upper bound of reasonable default (produces ~2000 DNA bases).
Run-length encoding (RLE)
Compressing a row like "BBBBBWWWWWWWWBB" as "5 black, 8 white, 2 black" — a list of run lengths instead of pixel-by-pixel. Works brilliantly for binary images with long runs.
Golomb-Rice
A simple binary code for small positive integers: a short one packs into few bits, a long one packs into more. Matches the statistics of image run lengths (many short, few long). We use this to turn the RLE numbers into a bit-string.
GR_K
The single parameter of Golomb-Rice. Set to 5 in this app. It tunes the code for the expected range of run-length values — fine for our typical images.
Salt search
Before accepting a chunk's DNA, the encoder tries up to 65,536 different salt values and keeps the first one whose DNA is bio-safe. Cheap because XOR is instant.
Bio-safety
A DNA is "bio-safe" here if its GC% sits between 40% and 60% and it contains none of a list of 12 restriction sites. This gives the strand a good chance of being synthesizable and surviving typical cloning workflows.
Restriction site
Short DNA sequences (usually 6 bases) that bacterial enzymes recognize and cut. If an important enzyme's site appears in your DNA, the enzyme will slice your strand during cloning. Commercial synthesis often also refuses sequences containing these.
GC content / GC%
The percentage of bases that are G or C (as opposed to A or T). Extreme values (too high or too low) make DNA harder to synthesize and sequence reliably. We aim for 40–60%.
